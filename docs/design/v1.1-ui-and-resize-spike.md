# winTerm 1.1 UI and pane-resize technical spike

## Baseline and integration boundary

winTerm is built on Microsoft Terminal revision
`1cea42d433253d95c4487a3037db48197b5e72f4`. The inherited `TerminalPage`,
`Tab`, and binary `Pane` tree remain the native runtime bridge. A branch
`Pane` owns two children, a vertical or horizontal orientation, and
`_desiredSplitPosition`; startup actions already serialize that ratio for
persisted-window restore. Workspace schema version 2 therefore needs no
persistent-model change.

The application shell is split across `TitlebarControl.xaml`,
`TabRowControl.xaml`, `TabHeaderControl.xaml`, `App.xaml`, and C++-constructed
pane visuals in `Pane.cpp`. The Windows host continues to own native caption
buttons, the system menu, double-click maximize, window resizing, and Windows
Snap. The terminal renderer and profile appearance pipeline remain separate
from the application-shell palette.

## Previous pane movement implementation

The pre-1.1 winTerm layer added a pane-handle drag source, target resolver,
Snap-style overlay, edge/corner/empty-slot destinations, drag session and
session ownership objects, a layout transaction, movement history, diagnostics,
keyboard movement mode, move actions, and pane-drag settings. Runtime glue in
`Pane`, `Tab`, `TabManagement`, and `TerminalPage` forwarded pointer movement
into that layer.

That stack is removed for 1.1. The winTerm build does not construct a pane
movement source or overlay, and its inherited `movePane` identifier is retained
only as a parse-safe disabled legacy action. The executable does not compile
the inherited pane-move implementation. Top-level tab reordering and tab
tear-out remain independent.

## Resize model

Each branch pane owns its divider. Pointer press captures the branch runtime
ID, original ratio, immediate child minimum sizes, and branch-local geometry.
Pointer movement clamps to valid geometry, evaluates snap points, and updates
only `_desiredSplitPosition` and the existing row or column definitions.
Release emits one committed ratio. Escape, cancellation, or pointer-capture
loss restores the original ratio.

The visible divider and pointer target are different objects: the line is one
logical pixel and the hit target is 12 logical pixels. The target uses the
appropriate east-west or north-south cursor. The resize code does not
serialize, write files, enumerate profiles, reload themes, rebuild windows, or
touch the terminal backend during pointer movement.

## Components preserved

- Existing pane split tree, minimum-size calculation, and terminal cell sizing
- Directed Top, Bottom, Left, and Right splitting
- Profile selection, default/current profile behavior, and rollback on failed
  shell creation
- Workspace split orientation, ratio, focus, tab, and window restore
- Top-level tab reorder and tear-out behavior
- Native caption behavior and Windows Snap
- Terminal renderer, ConPTY, VT parser, text buffer, Unicode width, input
  protocol, user profile themes, shell sessions, installer, and Portable mode

## Upstream conflict forecast

The highest-risk future upstream merges are `Pane.cpp`/`Pane.h`, action enum
generation, the Settings editor navigation, `TerminalPage` pane menus, and tab
header/titlebar XAML. Design-token and pane-resize model files are isolated
under `src/winterm` to keep conflict resolution narrow. No protected terminal
core area is modified.

## DPI, accessibility, and performance risks

- XAML dimensions and pointer coordinates are device-independent pixels, but
  mixed-DPI monitor transitions still require a real-window validation pass.
- A branch must recompute placement after layout/DPI changes so the invisible
  hit target stays centered on the visible divider.
- Minimum sizes include terminal cell constraints and pane-header height;
  invalid snap points must be filtered on every transaction.
- High Contrast must replace fixed divider colors with theme resources.
- The snap state needs a text/automation signal in addition to color without
  moving keyboard focus into terminal content.
- Continuous terminal resize can expose renderer latency under heavy output;
  Release-build frame timing and memory measurements remain acceptance gates.
